Why I Deleted a 0.996 Correlated Feature — And Why That Was Right

Preeti Bhardwaj  ·  May 2026

When I was building a bank customer churn prediction model on 10,000 records, my correlation matrix threw up something that made me stop and stare.

One feature — Complain (whether a customer had filed a complaint) — had a 0.996 correlation with the target variable (Exited). That's almost perfect. A data scientist's dream, right?

I deleted it anyway. Here's why.

What the number was telling me

A 0.996 correlation means: almost every customer who complained, churned. Almost every customer who didn't complain, stayed. On the surface, that sounds like a powerful signal. Your model would love it. Your AUC would look great.

But ask yourself — in the real world, when does a bank know a customer has complained?

Only after the customer has already complained. Which happens right before they leave. So what you've built is not a churn predictor. It's a churn confirmer. By the time you have the Complain value, the customer has one foot out the door. There's no time to intervene. A model like that is useless to a retention team.

The difference between correlation and usefulness

This is the thing tutorials don't teach you. High correlation with the target feels like a win. But features need to pass two tests, not one:

Test 1 — Is it statistically useful? (correlation, feature importance)
Test 2 — Is it available and actionable at prediction time?

Complain passed Test 1 with flying colours. It failed Test 2 completely. A churn model is only valuable if it gives the business time to act — send a retention offer, assign a relationship manager, trigger a call. For that, you need features available weeks or months before churn. Not features that appear because churn is happening.

What I kept instead

After removing Complain, the model had to find real, leading signals. SHAP analysis revealed the actual drivers:

  • Age — older customers churn more (possibly less digital-native, more likely to switch to traditional banks)
  • NumOfProducts — customers with only 1 product churn at much higher rates; those with 2–3 are stickier
  • Geography — Germany showed 32.4% churn vs ~16% in France and Spain, suggesting a market-specific problem

These are features a retention team can actually use. You can identify a 45-year-old single-product German customer today and reach out before they decide to leave.

The model still performed well

Removing a 0.996 correlated feature didn't ruin the model:

  • ROC-AUC: 0.847
  • Mean CV AUC: 0.87 across 5 folds

That's a solid, trustworthy model — built on features that make business sense, without data leakage risk, and without a feature that would be useless in production.

The lesson

When you see a suspiciously high correlation, don't celebrate. Ask:

"Would I actually have this value at the time I need to make a prediction?"

If the answer is no — remove it. A lower AUC built on honest features is worth more than a higher AUC built on a feature that tells you what you already know. ML isn't just about the metric. It's about building something that works in the real world.

Building Your First RAG Pipeline — No Fluff, Just Steps

Preeti Bhardwaj  ·  May 2026

Every tutorial I found when I started learning RAG was either too theoretical or assumed I already knew half the stack. This is the tutorial I wish I had — no fluff, no jargon walls, just the actual steps to build a working RAG pipeline from scratch.

What is RAG and why does it matter?

RAG = Retrieval-Augmented Generation. LLMs like GPT or Gemini don't know your documents. If you ask "What does page 14 of this medical report say?" they'll hallucinate an answer. RAG fixes this: it takes your documents, turns them into searchable chunks, finds the most relevant ones for your question, and sends them to the LLM as context. The LLM answers from your documents, not from training data.

What we're building

A pipeline where you upload any PDF, ask questions in plain English, and get accurate answers. Stack: Python · LangChain · FAISS · Google Gemini (free API)

Step 1 — Set up your environment

pip install langchain langchain-google-genai faiss-cpu pypdf sentence-transformers

Get a free Gemini API key at aistudio.google.com — no credit card needed.

Step 2 — Load your document

from langchain.document_loaders import PyPDFLoader

loader = PyPDFLoader("your_document.pdf")
documents = loader.load()

Step 3 — Split into chunks

from langchain.text_splitter import RecursiveCharacterTextSplitter

splitter = RecursiveCharacterTextSplitter(
    chunk_size=500,
    chunk_overlap=50
)
chunks = splitter.split_documents(documents)

The overlap is important — it prevents sentences at chunk boundaries from losing context.

Step 4 — Create embeddings and store in FAISS

from langchain.embeddings import HuggingFaceEmbeddings
from langchain.vectorstores import FAISS

embeddings = HuggingFaceEmbeddings(model_name="all-MiniLM-L6-v2")
vectorstore = FAISS.from_documents(chunks, embeddings)

all-MiniLM-L6-v2 is free, fast, and runs locally — no API calls for this step.

Step 5 — Set up Gemini

from langchain_google_genai import ChatGoogleGenerativeAI

llm = ChatGoogleGenerativeAI(
    model="gemini-pro",
    google_api_key="YOUR_API_KEY"
)

Step 6 — Build the RAG chain

from langchain.chains import RetrievalQA

qa_chain = RetrievalQA.from_chain_type(
    llm=llm,
    chain_type="stuff",
    retriever=vectorstore.as_retriever(search_kwargs={"k": 3})
)

Step 7 — Ask questions!

question = "What are the main risk factors mentioned in this document?"
answer = qa_chain.run(question)
print(answer)

That's it. Your RAG pipeline is working.

Common mistakes to avoid

  • Chunk size too large — start with 500 characters
  • No overlap — always use ~10% overlap
  • Using LLM for embeddings — use MiniLM locally, save API quota for generation
  • Not testing retrieval separately — verify your retriever finds the right chunks before testing the full chain

Where to go from here

Once basics work: add a Streamlit UI for file uploads, add conversation memory, try parent-child chunking for long documents, add source citation showing which page the answer came from.

RAG is one of the most practical GenAI skills you can have right now. Build it once, understand it deeply, and you'll be ahead of most people calling themselves GenAI developers.
